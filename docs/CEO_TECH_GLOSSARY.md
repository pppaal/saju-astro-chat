## 1. 기본 구조

### 🏪 카페로 이해하는 웹 서비스

| 용어         | 카페 비유                  | 실제 의미               | 예시                      |
| ------------ | -------------------------- | ----------------------- | ------------------------- |
| **Frontend** | 카페 홀 (고객이 보는 공간) | 웹사이트 화면, 디자인   | 버튼, 색상, 레이아웃      |
| **Backend**  | 주방 (뒤에서 일하는 곳)    | 서버, 데이터 처리       | 사주 계산, 결제 처리      |
| **Database** | 창고 (재료 보관)           | 데이터 저장소           | 고객 정보, 결제 내역      |
| **API**      | 주문 벨 (주방↔홀 연결)     | 데이터 주고받는 통로    | "사주 데이터 주세요" 요청 |
| **Cache**    | 미리 만든 샌드위치         | 자주 쓰는 데이터 빠르게 | 오늘의 운세 (12시간 저장) |
| **Server**   | 카페 건물 전체             | 서비스 돌아가는 컴퓨터  | Vercel, AWS 같은 곳       |
| **Client**   | 손님 (브라우저)            | 사용자 기기             | 크롬, 사파리              |
| **Request**  | 주문 ("카페라떼 주세요")   | 데이터 요청             | "사주 보기" 클릭          |
| **Response** | 커피 배달                  | 서버 응답               | 사주 결과 화면            |

## 3. AI & 머신러닝

### AI 모델 (직원 등급)

| **추가 공급자(후보)**   | 다른 카페 마스터       | 외부 LLM             | 💰💰          | 백업용                 |

| **Together AI**     | 협력 카페 2            | AI 클라우드 서비스   | 💰💰          | 2차 백업               |

### AI 기술 용어

| 용어               | 쉬운 비유                 | 실제 의미                | 예시                     |
| ------------------ | ------------------------- | ------------------------ | ------------------------ |
| **Prompt**         | 주문서 ("아메리카노 1잔") | AI 지시문                | "사주를 분석해줘"        |
| **Token**          | 커피콩 1알 (재료 단위)    | AI 계산 단위             | 한글 1글자 ≈ 2-3 토큰    |
| **Context Window** | 바리스타 기억력           | AI가 한 번에 기억하는 양 | 모델별 상이 (예: 128k)   |
| **RAG**            | 레시피북 참고             | 전문 지식 주입           | 사주 이론 1,000개 문서   |
| **Embedding**      | 재료 바코드               | 텍스트→숫자 변환         | AI가 의미 이해           |
| **Fine-tuning**    | 바리스타 전문 교육        | AI 맞춤 학습             | 사주 전문가로 훈련       |
| **Temperature**    | 창의성 온도               | AI 창의성 조절           | 0=정확, 1=창의적         |
| **Streaming**      | 조금씩 따라주기           | 실시간 응답              | 타로 해석 글자씩 출력    |
| **Fallback**       | 백업 바리스타             | 대체 시스템              | OpenAI 안 되면 Replicate |

### AI 비용 관리

| 용어                  | 쉬운 비유        | 실제 의미        | 우리 전략                    |

### 데이터 구조

| 용어            | 창고 비유           | 실제 의미        | 예시                   |
| --------------- | ------------------- | ---------------- | ---------------------- |
| **Schema**      | 창고 설계도         | 데이터 구조 설계 | User, Post, Payment    |
| **Model**       | 물건 종류           | 데이터 타입      | User, SajuResult       |

| **Index**       | 창고 색인 (책 뒷면) | 빠른 검색용      | 이메일로 빠르게 찾기   |
| **Migration**   | 창고 리모델링       | DB 구조 변경     | "생년월일 컬럼 추가"   |
| **Seed**        | 초기 재고           | 테스트 데이터    | 더미 사용자 10명       |

## 5. 인증 & 보안

### 보안 기술

| 용어           | 금고 비유     | 실제 의미       | 왜 중요한가            |

## 7. 성능 & 최적화

### 속도 개선

| 용어               | 자동차 비유      | 실제 의미                | 목표             |
| ------------------ | ---------------- | ------------------------ | ---------------- |

### RAG (Retrieval-Augmented Generation) - 검색 증강 생성

**왜 필요한가:**
AI(ChatGPT)한테 "사주 봐줘"하면 대충 아는 척하면서 틀린 말을 한다.
사주 전문 지식이 학습 데이터에 충분하지 않기 때문이다.

**작동 방식:**

```
사용자: "1990년 3월 15일생 사주 봐줘"
       ↓
1단계: 사주 전문 자료에서 관련 내용 검색 (벡터DB + 키워드 검색)
       ↓
2단계: 검색 결과 중 가장 관련 높은 것 순위 매기기
       ↓
3단계: 여러 검색 결과를 하나로 합치기 (퓨전)
       ↓
4단계: AI가 검색된 자료를 참고해서 답변 생성
       ↓
사용자에게 전문적인 사주 분석 결과 전달
```

**한 줄 설명:**
"AI가 답하기 전에 전문 자료를 먼저 찾아보고 그걸 참고해서 답하는 방식"

**이 프로젝트에서:**
8개 도메인(사주, 타로, 주역, 점성술, 꿈해몽, 수비학, 전생, 궁합)마다
각각 별도의 RAG 파이프라인을 구축했다.

---

### 벡터DB / ChromaDB - 의미 기반 검색

**일반 DB 검색 vs 벡터DB 검색:**

```
일반 DB: "천간"을 검색하면 → "천간"이라는 글자가 있는 것만 나옴
벡터DB: "천간"을 검색하면 → "갑을병정무기경신임계"도 나옴 (의미가 관련있으니까)
```

**원리:**

```
모든 텍스트를 숫자 배열(벡터)로 변환해둠:

"갑목은 봄의 기운이다"    → [0.23, 0.87, 0.12, 0.45, ...]
"을목은 부드러운 나무다"  → [0.25, 0.85, 0.14, 0.43, ...]  ← 숫자가 비슷함!
"오늘 날씨가 좋다"        → [0.91, 0.03, 0.77, 0.11, ...]  ← 완전 다른 숫자

숫자가 비슷한 것 = 의미가 비슷한 것 → 이걸로 검색
```

**한 줄 설명:**
"텍스트를 숫자로 바꿔서 '의미가 비슷한 것'을 찾아주는 데이터베이스"

---

### 하이브리드 검색 - 두 가지 검색을 합친 것

```
벡터 검색만 쓰면: 의미적으로 비슷한 건 찾지만, 정확한 단어 매칭을 놓칠 수 있음
키워드 검색(BM25)만 쓰면: 정확한 단어는 찾지만, 다른 표현으로 쓴 건 못 찾음
둘 다 합치면: 정확한 키워드 매칭 + 의미적 연관성 둘 다 잡음
```

**한 줄 설명:**
"키워드 검색(정확한 단어)과 벡터 검색(비슷한 의미)을 합쳐서 검색 정확도를 높인 것"

### 미들웨어 - 공통 관문

```
미들웨어 없이 135개 API를 만들면:
→ 각 API마다 로그인 확인 코드를 따로 써야 함
→ 각 API마다 보안 검사를 따로 써야 함
→ 135번 같은 코드를 반복
→ 하나라도 빠뜨리면 보안 구멍

미들웨어를 만들면:
→ 공통 코드 한 번만 작성
→ 135개 API가 전부 동일한 보안 검사를 거침
→ 빠뜨릴 일이 없음
```

건물 입구에 경비실 하나를 두는 것과 같다.
모든 방에 경비원을 따로 두는 게 아니라, 입구에서 한 번에 검사한다.

**이 프로젝트의 미들웨어가 하는 일:**

1. 로그인 했는지 확인
2. 입력값이 올바른지 검증
3. 같은 사람이 너무 많이 요청하면 차단 (Rate Limiting)
4. 크레딧 잔액 확인 (유료 기능일 때)
5. 에러 발생 시 표준화된 응답
6. 모든 요청을 로그에 기록

---

### CI/CD 상세 - 자동 검사 + 자동 배포

```
내가 코드를 수정해서 GitHub에 올림
       ↓
[CI - 자동 검사] (GitHub Actions가 알아서 실행)
  ✅ 코드 문법 검사 (린트)
  ✅ 타입 오류 검사 (TypeScript)
  ✅ 테스트 813개 실행
  ✅ 보안 취약점 스캔
  ✅ 빌드 성공하는지 확인
       ↓
  전부 통과하면
       ↓
[CD - 자동 배포]
  → Vercel에 프론트엔드 배포
  → Fly.io에 AI 백엔드 배포
       ↓
  사용자가 바로 새 버전을 볼 수 있음
```

**이 프로젝트의 13개 워크플로우:**

1. 기본 검사 (린트, 타입, 테스트, 빌드)
2. 프로덕션 배포
3. 미리보기 배포 (PR 올리면 테스트용 사이트 자동 생성)
4. 백엔드 배포
5. 보안 검사 (토큰 암호화 확인, 환경변수 검증)
6. 브라우저 E2E 테스트
7. 접근성 검사 (장애인도 쓸 수 있는지)
8. OWASP 보안 스캔
9. 부하 테스트
   10~13. PR 검사, 품질 검사, 마케팅 자동화 등

**핵심:**
혼자 개발하면 실수를 잡아줄 동료가 없으니까, 이 자동 검사가 동료 역할을 한다.

### TypeScript strict 모드 + Zod - 이중 안전장치

**TypeScript strict 모드:**

```
JavaScript:   let age = "스물다섯"   ← 나이에 글자를 넣어도 에러 안 남
              age + 1 = "스물다섯1"  ← 나중에 이상한 결과가 나옴

TypeScript:   let age: number = "스물다섯"  ← 바로 에러! "숫자를 넣어야 해"

strict 모드: 이 검사를 가장 엄격하게 하는 설정
```

**Zod:**

```
사용자가 API로 데이터를 보낼 때:

정상: { name: "홍길동", birthDate: "1990-03-15" }  ← OK
비정상: { name: 123, birthDate: "어제" }            ← Zod가 차단

Zod가 하는 일:
→ "name은 문자열이어야 합니다" ❌
→ "birthDate는 YYYY-MM-DD 형식이어야 합니다" ❌
→ 잘못된 데이터가 서버로 들어가기 전에 차단
```

**차이:**

- TypeScript: 코드를 쓸 때 (개발 중에) 타입 오류를 잡아줌
- Zod: 서비스가 돌아갈 때 (실제 사용 중에) 잘못된 데이터를 잡아줌
- 둘 다 필요함 = 이중 안전장치

### 보안 용어 심화

**AES-256-GCM 상세:**

```
카카오 로그인하면 카카오에서 토큰(열쇠)을 줌

암호화 안 하면:
DB에 저장: "kakao_token_abc123xyz"
→ DB 털리면 이 토큰으로 카카오 계정 접근 가능

암호화 하면:
DB에 저장: "x8fK2mP9qL3nR7..."  (알아볼 수 없는 문자)
→ DB 털려도 원래 값을 알 수 없음

256 = 암호 키 길이 (256비트). 현존 컴퓨터로 풀려면 우주 나이보다 오래 걸림
GCM = 암호화 모드. 데이터가 중간에 변조됐는지도 감지해줌
```

**CSRF 상세:**

```
시나리오:
1. 내가 운세 사이트에 로그인한 상태
2. 해커가 만든 사이트를 클릭
3. 그 사이트가 몰래 "100 크레딧 결제해줘"라고 우리 서버에 요청
4. 서버는 로그인된 사용자의 요청이니까 처리해버림

방어법:
→ 모든 요청에 "어디서 보낸 거야?" 확인 (Origin/Referer 검증)
→ 우리 사이트에서 보낸 게 아니면 차단
```

**Rate Limiting 상세:**

```
같은 IP에서 1분에 100번 요청이 오면:
→ 정상 사용자가 아님 (공격이거나 자동화 프로그램)
→ 자동 차단

Redis에 기록:
IP 123.456.789 → 이번 1분간 요청 횟수: 47
IP 123.456.789 → 이번 1분간 요청 횟수: 48
...
IP 123.456.789 → 이번 1분간 요청 횟수: 100 → 차단!
```

**OWASP:**

```
웹 보안 취약점 Top 10을 정리한 국제 기준
1위: 인증 실패
2위: 암호화 실패
3위: 인젝션 (SQL 주입 등)
...
"OWASP 기준으로 접근했다" = "국제 보안 기준에 맞춰 개발했다"
```

### Circuit Breaker - 회로 차단기 (약점 설명용)

외부 서비스(OpenAI 등)가 고장났을 때의 보호 장치:

```
Circuit Breaker 없으면 (현재 이 프로젝트 상태):
→ 우리 서버가 OpenAI 응답을 계속 기다림
→ 기다리는 요청이 쌓임
→ 우리 서버도 같이 죽음 (연쇄 장애)

Circuit Breaker 있으면:
→ "OpenAI가 3번 연속 실패했으니 잠시 차단"
→ 우리 서버는 바로 "지금 서비스 불가" 응답
→ 우리 서버는 살아있음
→ 일정 시간 후 다시 시도 → 복구되면 정상 운영
```

전기 차단기(Breaker)처럼 과부하가 걸리면 자동으로 끊어서 전체 시스템을 보호하는 패턴.
이 프로젝트에는 아직 미적용 → 개선할 점으로 인식하고 있음.

## 17. Graph RAG 아키텍처 용어 사전

> 면접에서 "백엔드 AI 시스템이 어떻게 동작하나요?" 질문에 답하기 위한 섹션

---

### 3줄 요약 (가장 쉬운 설명)

| 비유 | 설명 |
| --- | --- |
| **일반 검색 (Vector RAG)** | 도서관에서 키워드로 책 찾기 |
| **Graph RAG** | 위키피디아처럼 링크 타고 관련 지식 따라가기 |

**이 시스템이 하는 일:**

1. 사주/점성술 지식을 그래프(노드+엣지)로 저장 — "갑목은 태양과 비슷하다" 같은 관계들
2. 사용자 질문이 오면 그래프를 탐색해서 관련 지식을 찾음
3. 찾은 지식을 LLM(GPT/Claude)에 넘겨서 답변 생성

---

### 기본 개념: Graph RAG가 뭔가?

**일반 RAG**: 질문이 들어오면 → 자료실에서 비슷한 문서를 찾아서 → AI에게 읽혀줌
**Graph RAG**: 질문이 들어오면 → 관계 지도(그래프)를 따라가며 연결된 지식을 찾아서 → AI에게 읽혀줌

```
일반 RAG:  "갑목" 검색 → "갑목 설명" 문서 1개 찾음

Graph RAG: "갑목" 검색 → 갑목과 연결된 것들을 줄줄이 찾음
           갑목 --비슷함-→ 태양(Sun)
           갑목 --상생--→ 병화
           갑목 --상극--→ 경금
           갑목 --퓨전--→ 목성(Jupiter)
           → 이 전체를 AI에게 넘김 → 훨씬 풍부한 답변
```

**왜 Graph RAG인가:**
사주/점성술은 개념끼리 관계가 핵심이다.
"갑목"만 알면 소용없고, 갑목이 뭐랑 맞고(상생), 뭐랑 안 맞는지(상극)를 알아야 한다.
그래프는 이런 관계를 저장하고 탐색하기에 최적의 구조다.

---

### Graph RAG 용어 하나씩 풀기

#### 노드 (Node) & 엣지 (Edge)

```
노드 = 동그라미 (개념 하나)
엣지 = 선 (관계 하나)

[갑목] ----상생---→ [병화]
  ↑ 노드             ↑ 노드
         ↑ 엣지

이 프로젝트에서:
- 노드 = 사주 개념(갑목, 을목...), 행성(태양, 달...), 타로 카드, 주역 괘 등
- 엣지 = "비슷하다", "상생이다", "상극이다", "같은 원소다" 등의 관계
- CSV 파일로 저장됨 (nodes_*.csv, edges_*.csv, relations_*.csv)
```

#### 임베딩 (Embedding)

```
컴퓨터는 "갑목"이라는 글자를 이해 못 함
그래서 숫자 리스트로 바꿔줌

"갑목은 봄의 기운"  → [0.23, 0.87, 0.12, 0.45, ...]
"을목은 부드러운 나무" → [0.25, 0.85, 0.14, 0.43, ...]  ← 비슷한 숫자!
"오늘 날씨 좋다"     → [0.91, 0.03, 0.77, 0.11, ...]  ← 전혀 다른 숫자

비슷한 의미 = 비슷한 숫자 → 이걸로 "의미가 비슷한 것"을 찾음

이 프로젝트에서:
- corpus_embeds.pt (110MB) = 사주/점성술 지식 문서 임베딩
- graph_rag_embeds.pt (3.6MB) = 그래프 노드 임베딩
- SentenceTransformer 모델이 텍스트 → 숫자 변환을 담당
```

#### 코사인 유사도 (Cosine Similarity)

```
두 임베딩(숫자 리스트)이 얼마나 비슷한지 점수 매기기

1.0 = 완전 같음
0.0 = 전혀 관련 없음

"사주 운세" vs "사주 풀이" → 0.9 (비슷!)
"사주 운세" vs "피자 주문" → 0.1 (관련 없음)

이걸로 "질문과 가장 관련 있는 지식"을 찾아냄
```

#### top_k

```
검색 결과 중 상위 k개만 가져오겠다는 뜻

이 프로젝트 설정:
- GraphRAG: top_k=15 (그래프에서 15개)
- CorpusRAG: top_k=6 (문서에서 6개)
- PersonaRAG: top_k=5 (페르소나 5개)
- DomainRAG: top_k=5 (도메인 지식 5개)
→ 총 31개 결과를 AI에게 넘김
```

---

### Fusion Layer (퓨전 레이어) - 동서양 연결

```
이 프로젝트의 핵심 차별점

사주(동양)와 점성술(서양)은 원래 별개의 체계인데,
이걸 3단계로 연결하는 매핑 테이블을 만들었다:

Layer 1 (기본 매핑) - 직관적 대응:
  목(木) ↔ Earth(땅)    → "나무는 땅에서 자란다"
  화(火) ↔ Fire(불)     → "같은 불"
  수(水) ↔ Water(물)    → "같은 물"
  갑(甲) ↔ Sun(태양)    → "갑목 = 우두머리 = 태양"

Layer 2 (기능 매핑) - 역할이 비슷한 것끼리:
  편관 ↔ 화성(Mars)     → 둘 다 "도전, 압박" 의미
  식신 ↔ 달(Moon)       → 둘 다 "표현, 감성" 의미
  삼합 ↔ Trine          → 둘 다 "조화로운 관계" 의미

Layer 3 (심층 매핑) - 영적/원형적 연결:
  공망 ↔ North Node     → 둘 다 "영혼의 과제" 의미
  역마 ↔ South Node     → 둘 다 "운명적 여정" 의미
  상관견관 ↔ Chiron     → 둘 다 "상처 입은 치유자" 의미

면접용 한마디:
"동양과 서양의 점술 체계를 3계층으로 매핑하여
단일 그래프에서 통합 분석이 가능하게 했습니다"
```

### HyDE (Hypothetical Document Embeddings) - 가설적 문서 임베딩

```
문제:
사용자가 "내 사주 어때?" 같이 짧게 물으면
검색이 잘 안 됨 (너무 짧고 모호해서)

해결법:
1. "내 사주 어때?" 를 AI에게 먼저 보냄
2. AI가 "가짜 답변"을 만들어냄:
   "갑목 일간이시군요. 목의 기운이 강하고
    올해 대운이 화(火)로 흐르니 식신의 에너지가..."
3. 이 긴 가짜 답변으로 검색함
4. 짧은 질문보다 훨씬 관련 높은 결과가 나옴

비유: 시험 볼 때
→ "경제" 로 교과서 찾기 (너무 광범위)
→ "GDP 성장률과 인플레이션의 상관관계" 로 찾기 (정확)
→ HyDE가 짧은 키워드를 구체적인 문장으로 바꿔주는 것

LLM 안 되면: 도메인 키워드를 붙여서 확장
"내 사주 어때?" → "내 사주 어때? 사주 오행 천간 지지 십신 일간 대운 세운"
```

---

### CrossEncoder 재순위화 (Reranker)

```
검색은 2단계로 한다:

1단계 - Bi-Encoder (빠르지만 대략적):
  질문과 문서를 각각 따로 숫자로 바꿈
  숫자끼리 비교해서 후보 45개를 뽑음
  → 빠름 (문서마다 0.001초)
  → 대신 좀 부정확할 수 있음

2단계 - CrossEncoder (느리지만 정확):
  질문+문서를 한 쌍으로 묶어서 동시에 판단
  45개 후보 중 진짜 관련 있는 15개만 남김
  → 느림 (문서마다 0.01초)
  → 대신 정확도가 15-25% 올라감

비유:
1단계 = 이력서 스크리닝 (빠르게 걸러냄)
2단계 = 실제 면접 (꼼꼼하게 평가)

이 프로젝트에서 50-100ms 추가 지연으로 정확도 15-25% 향상
```

### Semantic Chunking - 의미 단위 나누기

```
문제: 긴 문서를 AI에 넣으려면 잘라야 함

나쁜 방법 (고정 크기):
  500자마다 기계적으로 자름
  → "갑목은 봄의 기운을 가진 천간으로, 양"  ← 여기서 잘림
  → "의 목(木)에 해당한다. 성격이 곧고..."   ← 문맥이 끊김

좋은 방법 (의미 단위):
  인접한 문장들의 임베딩 유사도를 비교
  유사도가 뚝 떨어지는 곳 = 주제가 바뀌는 곳
  거기서 자름
  → "갑목은 봄의 기운을 가진 천간으로, 양의 목(木)에 해당한다."  ← 의미 완결
  → "을목은 부드러운 음의 나무로..."  ← 새 주제 시작

한국어 + 영어 문장 분리 모두 지원
```

### 그래프 알고리즘 (쉽게)

#### PageRank

```
구글이 웹페이지 순위를 매기는 방법을 그래프에 적용

원리: 다른 노드들이 많이 가리키는 노드 = 중요한 노드

예시:
"오행" 노드에 연결된 엣지가 50개 → 매우 중요한 개념
"특수신살X" 노드에 연결된 엣지가 2개 → 덜 중요한 개념

용도: 검색 결과 중 중요한 개념을 우선 보여줌
```

#### Community Detection (커뮤니티 탐지)

```
그래프에서 자연스러운 그룹을 찾아냄

예시:
사주 관련 노드들끼리 많이 연결됨 → "사주 커뮤니티"
점성술 관련 노드들끼리 많이 연결됨 → "점성술 커뮤니티"
퓨전 노드들이 두 커뮤니티를 연결 → "다리 역할"

용도: "이 질문은 사주 쪽이구나" 판단에 활용
```

#### Beam Search (빔 서치)

```
그래프를 탐색할 때 모든 길을 다 가보면 너무 오래 걸림

Beam Search: 유망한 길 k개만 따라감

예시 (k=3):
"갑목"에서 출발 → 연결된 노드 20개
→ 점수 높은 3개만 선택: 태양, 병화, 목성
→ 각각에서 또 연결된 노드 중 상위 3개만...
→ 빠르게 관련 지식에 도달

모든 길: 20 × 20 × 20 = 8,000개 탐색
Beam(k=3): 3 × 3 × 3 = 27개만 탐색
```

### OptimizedRAGManager - 총괄 매니저

```
4개의 RAG를 동시에 관리하는 매니저

[질문 들어옴]
    ↓
캐시에 있나? → 있으면 바로 응답 (5분 이내 같은 질문)
    ↓ 없으면
[HyDE로 질문 확장] (선택 사항)
    ↓
[4개 RAG 동시 검색] ← ThreadPoolExecutor (일꾼 4명)
  ├→ GraphRAG:  그래프 탐색 (15개)
  ├→ CorpusRAG: 문서 검색 (6개)
  ├→ PersonaRAG: 페르소나 (5개)
  └→ DomainRAG: 도메인 지식 (5개)
    ↓
[CrossEncoder 재순위화] (선택 사항)
    ↓
[결과 합쳐서 AI에게 전달]

성능:
  최적화 전: 1,500ms (1.5초)
  최적화 후: 300-500ms (0.3-0.5초)
  → 약 70% 빨라짐
```

### 캐싱 전략 (유통기한별 정리)

```
모든 데이터에 유통기한(TTL)이 있다:

사주 데이터: 7일  ← 태어난 날은 안 바뀌니까
점성술 운세: 24시간 ← 행성 위치가 매일 바뀌니까
타로/꿈해몽: 48시간 ← 리딩 결과가 이틀 정도는 유효
불변 규칙: 30일  ← "갑목 = 양의 나무" 같은 건 안 바뀜
RAG 검색: 5분   ← 자주 바뀔 수 있으니 짧게

Redis 죽으면? → 자동으로 인메모리 캐시(서버 메모리)로 전환
이게 Circuit Breaker 패턴
```

### RAGAS 평가 - RAG 성적표

```
"우리 RAG가 잘 하고 있나?" 를 점수로 매기는 프레임워크

5개 항목:

1. Faithfulness (신뢰성)
   → 답변이 근거 없는 말을 하진 않는지
   → "갑목이 금의 기운입니다" ← 이런 거 하면 감점

2. Answer Relevancy (답변 관련성)
   → 질문에 맞는 답을 하고 있는지
   → "사주 봐줘" 했는데 타로 얘기하면 감점

3. Context Recall (검색 완전성)
   → 필요한 정보를 빠뜨리지 않았는지
   → 갑목 설명에서 상생 관계를 빠뜨리면 감점

4. Context Precision (검색 정확성)
   → 쓸데없는 정보를 안 가져왔는지
   → 갑목 질문에 타로 카드 정보까지 가져오면 감점

5. Entity Recall (핵심 용어 추출)
   → 사주 용어(천간, 지지, 오행)를 잘 잡아냈는지

26개 평가 문제가 내장되어 있음 (사주, 점성술, 타로, 퓨전 등)
LLM 평가 모드 + 로컬(무비용) 평가 모드 지원
```

### Tracing (추적) - 파이프라인 관찰

```
RAG 파이프라인의 모든 단계를 기록

기록하는 것:
- search: 각 RAG 소스별 검색 시간, 결과 점수
- rerank: 재순위화 전후 결과 수
- hyde: 가설 생성에 걸린 시간
- llm: AI 응답 토큰 수, 생성 시간
- error: 에러 발생 시 상세 정보

3곳에 동시 기록:
1. 로컬 JSON 파일 (항상)
2. LangSmith (설정 시) - LangChain 공식 모니터링
3. Langfuse (설정 시) - 오픈소스 모니터링

용도: "왜 이 답변이 느렸지?" "왜 이상한 답이 나왔지?" 추적
```

### Monitoring (모니터링) - 시스템 감시

```
서버 전체를 실시간으로 감시

경고 기준:
- 응답시간 700ms 초과 → 경고
- 에러율 0.5% 초과 → 경고
- 메모리 450MB 초과 → 경고

측정하는 것:
- 엔드포인트별 요청 수
- 평균/최소/최대 응답 시간
- 에러 수 및 마지막 에러 내용
- 가장 느린 엔드포인트 순위

건강 상태: healthy / degraded / critical
```

### 면접에서 이렇게 설명하세요

**"Graph RAG 시스템을 간단히 설명해주세요"**

> "사주와 점성술 지식을 그래프 구조로 저장합니다. 노드는 개념(갑목, 태양 등),
> 엣지는 관계(상생, 상극, 비슷함)입니다. 사용자 질문이 오면 그래프를 탐색해서
> 연결된 지식을 찾고, 4개 RAG 소스를 병렬로 검색해서 AI에 컨텍스트로 넘깁니다."

**"왜 일반 RAG 안 쓰고 Graph RAG를 썼나요?"**

> "사주와 점성술은 개념 간 관계가 핵심입니다. 갑목을 알려면 상생/상극/합충 관계를
> 함께 알아야 합니다. 일반 벡터 검색은 독립된 문서만 찾지만, 그래프 탐색은
> 관계를 따라 다중 홉 추론이 가능합니다."

**"Fusion Layer가 뭔가요?"**

> "사주(동양)와 점성술(서양)은 별개의 체계인데, 이걸 3계층으로 매핑했습니다.
> Layer 1은 기본(목↔Earth), Layer 2는 기능(편관↔화성),
> Layer 3은 영적 대응(공망↔North Node)입니다.
> 이렇게 하면 동서양 통합 분석이 가능합니다."

**"성능 최적화는 어떻게 했나요?"**

> "4가지입니다. 첫째, 4개 RAG를 ThreadPoolExecutor로 병렬 검색.
> 둘째, TTL LRU 캐시로 반복 질문 즉시 응답.
> 셋째, 사전 임베딩 로딩으로 콜드스타트 제거.
> 넷째, Circuit Breaker로 외부 서비스 장애 시 자동 전환.
> 결과적으로 p95 1500ms에서 700ms 이하로 개선했습니다."

**"RAG 품질은 어떻게 측정하나요?"**

> "RAGAS 프레임워크 5개 메트릭으로 측정합니다.
> Faithfulness(답변 신뢰성), Relevancy(관련성), Context Recall(검색 완전성),
> Context Precision(검색 정확성), Entity Recall(핵심 용어 추출).
> 26개 도메인 특화 평가셋이 내장되어 있고, LLM 평가와 로컬 평가 두 모드를 지원합니다."
